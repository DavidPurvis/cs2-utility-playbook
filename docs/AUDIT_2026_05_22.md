## Severity legend

- **CRITICAL** — user-visible bug, observable today, likely to cause confusion / wrong behavior on first encounter.
- **HIGH** — visible bug at lower frequency OR latent risk that will fire under common future changes.

## CRITICAL findings (5)

### C-1. All 5 scenarios are functionally empty
- **File:** `src/data/dust2.json` lines 556–720
- **Evidence:** verified — every player in every scenario has `actions: []`. Zero scenarios have any playable content.
- **User impact:** opening Scenarios tab shows 5 cards. Clicking any card opens scenario detail with role tabs and an empty action list saying "Pick a role above to see that player's chronological lineups." Picking any role shows the SAME empty state. The headline feature ("let's do scenario 4") has nothing inside.
- **Note:** code comment in `loadDust2.ts` acknowledges this is intentional seed state. But for a shipped product, scenarios with zero actions are a dead end. Either populate them or remove them from the seed.

### C-2. 7 of 10 lineups are orphan / unreachable from scenarios + CT guide
- **File:** `src/data/dust2.json`
- **Orphans:** `xbox_smoke`, `a_ct_smoke`, `a_long_flash`, `a_short_flash`, `b_tunnel_flash`, `b_window_smoke`, `ct_molly_from_long`
- **Reachable lineups (referenced anywhere):** only 3 (referenced via ctPositions[].recommendedLineupIds)
- **Nuance:** the orphans ARE reachable via the Map tab (origin-first navigation surfaces all lineups via throw-from markers) and via Instant Smokes (if within 1500 wu of a spawn). They're not dead — but they exist only as flat lineups, not embedded in any tactical context.
- **Combined with C-1:** the app's Scenario flow has functionally NO content while the Map tab is the only meaningful surface. Inverts the URD's headline product purpose.

### C-3. `activeThrowFromKey` not cleared on navigation
- **File:** `src/reducer.ts` lines ~60–110
- **Repro:**
  1. Home → Map tab
  2. Click any cluster marker → marker highlights, side panel populates
  3. Click Scenarios tab → marker state persists in reducer (just not visible)
  4. Click a scenario card → SELECT_SCENARIO, view → "scenario"
  5. Click back → view → "home"
  6. Click Map tab again
  7. **Expected:** no marker highlighted (fresh state)
  8. **Actual:** the original marker is still highlighted from step 2
- **Fix shape:** SELECT_SCENARIO, BACK (from scenario), and GO_HOME should reset `activeThrowFromKey: null`.

### C-4. CLI silently defaults `landingAt` to `{percent: {x:50, y:50}}` on parse failure
- **File:** `scripts/new-lineup.mjs` line ~183
- **Repro:** `npm run new-lineup -- --landing "setpos garbage"` — does NOT error; silently uses radar-center as fallback.
- **Violates:** URD anti-requirement AR-1 "do not invent data."
- **User impact:** if owner makes a typo in the `--landing` flag, a lineup gets created with bogus radar-center coordinates. Boot validator accepts it (it has a valid landingAt.percent). The lineup renders at the center of the map regardless of where it actually lands.
- **Fix shape:** CLI must error and exit non-zero on landing-parse failure.

### C-5. Spawn dot touch target ~7×7 px on mobile (375px viewport)
- **File:** `src/components/SpawnPicker.tsx` line ~134 (r=0.95 viewBox units constant)
- **Math:** at 375px viewport, single-column app-grid → radar ~343×343 px → cluster viewBox ~40 units → 0.95 vbu × (343/40) = **8.1 px diameter** for unpicked dots.
- **WCAG 2.5.5:** minimum touch target is 44×44 px (24×24 with sufficient spacing). At 8 px, dots are 5× smaller than the minimum.
- **Tension with the spawn-picker overlap fix:** the dot was deliberately shrunk to prevent overlap-stealing (DECISIONS_LEDGER R-12). Making the visible dot larger would re-introduce overlap; making the hit zone larger (invisible) re-introduces overlap-stealing.
- **Possible resolution:** zoom the spawn-cluster viewBox tighter (smaller padding) so each dot RENDERS at 14+ px while keeping the same fraction-of-viewBox radius. Worth experimenting with — the current padding=7 may be excessive.

## HIGH findings (9)

### H-1. `SELECT_TAB` does not call `history.pushState` (browser back-button is broken across tabs)
- **File:** `src/App.tsx` line ~52
- **Repro:** start on Scenarios. Click Map tab. Click browser back. You expected to return to Scenarios → you get... whatever the previous URL was, OR a no-op. Tab state is invisible to the browser history stack.
- **Fix shape:** wrap `dispatch({type: "SELECT_TAB"})` with `history.pushState({view:"home", tab})` in the same pattern as SELECT_SCENARIO + SELECT_LINEUP + GO_HOME.

### H-2. MapTab setpos `<code>` block overflows on mobile
- **File:** `src/components/tabs/MapTab.tsx` lines ~212–224
- **Issue:** the `<code>` containing `throwFrom: setpos X Y Z` has no `white-space: pre-wrap` or `overflow-x: auto`. A typical setpos line is ~30–45 chars; on a 375px viewport, the aside panel is ~343px wide, the code block has ~16px padding, leaving ~327px for ~9-char-per-line monospace text → guaranteed horizontal overflow.
- **Why not caught by existing tests:** the Map tab snapshot tests on mobile WERE added in PR #21, but the active-marker snapshot was taken with a marker whose setpos is shortish ("setpos -299.969 -1163.764" = ~25 chars). A marker with 4-digit coords (common on a fully-populated map) would overflow.

### H-3. No ref integrity for `defaults.spawnRushes` spawn IDs
- **File:** `src/data/loadDust2.ts` (validator doesn't walk `defaults.spawnRushes`)
- **Tracked as:** DECISIONS_LEDGER W-12 (already documented as known)
- **Risk:** if a spawn is renamed or deleted, spawn-rush rows render with broken chip text (the rush table chip uses `spawnsById.get(id)?.label ?? id` — falls back to the raw id like `dust2-t-s99`).

### H-4. No ref integrity for `scenarios[].players[].startingSpawnId`
- **File:** `src/data/loadDust2.ts`
- **Same class of gap as H-3** but for scenarios. If a player references a non-existent `startingSpawnId`, the radar dot for that player just doesn't render (silent failure). Should throw at boot.

### H-5. Tab buttons missing `aria-controls`; tab panels missing `role="tabpanel"`
- **Files:** `src/components/TabBar.tsx` lines ~48–96 (no aria-controls); `src/components/Home.tsx` lines ~49–72 (no role="tabpanel" on content)
- **WAI-ARIA Tabs pattern incomplete.** Screen-reader users navigating tabs hear "tab" but don't get the linkage between tab and panel; assistive tech can't auto-jump.
- **Fix shape:** `<button role="tab" id="tab-{id}" aria-controls="panel-{id}" aria-selected={...}>` + `<div role="tabpanel" id="panel-{id}" aria-labelledby="tab-{id}">`.

### H-6. `accent` color (#C67C4E) fails WCAG-AA on cream
- **File:** `src/theme.ts` line ~33
- **Measured ratio:** 3.12:1 (AA requires 4.5:1 for normal text, 3:1 for large)
- **Used for:** button labels, link text, scenario number chips, active-state fills throughout
- **Impact:** users with reduced contrast sensitivity (including some autistic users with visual processing differences — relevant to our stated audience) may miss accent-colored actions.
- **Note:** `accentDk` (#A56235) is 4.53:1 — passes AA just barely. Consider promoting accentDk to the default and reserving accent for large decorative use only.

### H-7. `textMute` color (#8E887C) fails AA on cream
- **File:** `src/theme.ts` line ~36 — comment says "non-body only"
- **But used in:** `src/components/TabBar.tsx` line ~87 for the tab hint sub-text ("plant spots · timings · spawn rushes" etc.)
- **Measured ratio:** 3.35:1
- **Impact:** tab hints are exactly the kind of explanatory text the structure-craving audience leans on. They should pass AA.

### H-8. Esc handler doesn't guard against form-input focus
- **File:** `src/App.tsx` lines ~96–104
- **Current behavior:** any keydown=Escape in any non-home view dispatches BACK.
- **Future risk:** if a search input or CLI helper is added in scenario / lineup view, Esc will close the view instead of clearing the input (standard form expectation).
- **Fix shape:** `if (e.target instanceof HTMLInputElement || ...) return;` before dispatching.

### H-9. SpawnPicker internal `side` state can desync from `pickedSpawnId`
- **File:** `src/components/SpawnPicker.tsx` line ~44 (useState for side, NOT reducer)
- **Current safeguard:** lines 80–88 clear `pickedSpawnId` when user clicks the side toggle and the picked spawn belongs to the old side.
- **Gap:** the safeguard only fires on user click. If reducer-driven changes set `pickedSpawnId` to a CT spawn while SpawnPicker's local `side === "T"`, the dot isn't visible (cluster only shows T). The chip shows "Spawn: CT-3" but no dot lights up.
- **Repro:** contrived today (no code path mutates pickedSpawnId across side); but if `PICK_SPAWN` ever fires from outside the SpawnPicker (e.g., scenario detail's startingSpawnId could one day cross-link), this desyncs.
- **Fix shape:** lift `side` into the reducer or derive from `pickedSpawnId`.

## MEDIUM findings (14)

### M-1. Validator doesn't enforce `landingAt.percent` ∈ [0..100]
- **File:** `src/data/loadDust2.ts`
- Out-of-bound landings render off the visible radar. No boot error.

### M-2. Validator doesn't bound `defaults.plants[].percent`
- Same class as M-1, for plant spots on the Defaults tab.

### M-3. Validator doesn't enforce scenario.number uniqueness
- CLI rejects dupes (`new-scenario.mjs` ~line 147); JSON hand-edit doesn't. Boot accepts duplicate numbers silently.

### M-4. Validator doesn't enforce `lineup.id` snake_case format
- CLI enforces `/^[a-z][a-z0-9_]*$/`; boot validator accepts any string. JSON hand-edits could introduce IDs that break URL slugs or test selectors.

### M-5. `SELECT_TAB` action never tested
- **File:** `src/reducer.test.ts`
- Action defined but no test verifies its behavior. Specifically: SELECT_TAB while in scenario / lineup view — does it cleanly update only `activeTab` and leave view unchanged? Currently yes, but uncovered.

### M-6. `SELECT_THROW_FROM` action never tested
- Same as M-5. Coverage gap in reducer.test.ts.

### M-7. `onSelectTab` callback never exercised in home.test.tsx
- **File:** `src/components/__tests__/home.test.tsx` lines ~28–80
- Test passes `noop` as `onSelectTab` but never clicks a tab to verify the callback fires with the correct value. Tab-switch behavior is only proven by the E2E suite (home-tabs.spec.ts), not at the unit level.

### M-8. CopyButton + LineupChip undersized for touch
- **Files:** `src/components/CopyButton.tsx` ~58–69 (~22–24 px tall), `src/components/CtPositionGuide.tsx` ~127 (chips ~17 px tall)
- Below 44×44 px touch target minimum. Manifest on mobile + for users with reduced motor control.

### M-9. No `prefers-reduced-motion` support
- **File:** `src/components/Radar.tsx` lines ~100–136 (400ms viewBox animation)
- Users with vestibular disorders or motion sensitivity (including some autistic users — the stated audience) may find the spawn-cluster zoom disorienting. Should check `window.matchMedia('(prefers-reduced-motion: reduce)')`.

### M-10. Skip link target only exists on Scenarios tab
- **Files:** `index.html` ~line 121 (skip link `<a href="#lineup-list">`); `src/components/tabs/ScenariosTab.tsx` line ~34 (`id="lineup-list"`)
- On Defaults, Instant Smokes, and Map tabs — no `#lineup-list` element. Skip link is a no-op on 3 of the 4 tabs.

### M-11. Asymmetric BACK rule untested at App + popstate level
- **File:** `src/reducer.test.ts` covers the rule at reducer level. But the rapid-popstate case (user mashes browser-back during a viewBox animation) is uncovered. Reducer would receive multiple BACK dispatches with potentially-stale state snapshots.

### M-12. `fullPage` inconsistency in visual-snapshots.spec.ts
- **File:** `tests/e2e/visual-snapshots.spec.ts`
- `home-t-side`: `fullPage: false`; `home-ct-side`: `fullPage: true`; `scenario-detail`: `fullPage: false`; Map tab desktop default: `fullPage: false`; Map tab mobile: `fullPage: true`.
- The inconsistency means some snapshots capture only the viewport, others the entire scroll area. A real regression outside the viewport could slip past a `fullPage: false` snapshot.

### M-13. Visual snapshots missing for Defaults + Instant Smokes tabs
- **File:** `tests/e2e/visual-snapshots.spec.ts`
- 7 of 7 current snapshots — none cover Defaults or Instant Smokes. Tracked as partial-W-11 in DECISIONS_LEDGER.

### M-14. E2E suite not in `npm run validate`
- **File:** `package.json` line ~25
- `validate` runs typecheck + lint + vitest + node:test + build, but NOT Playwright E2E. The E2E suite catches visual regressions and click-precision bugs that vitest can't see. It's run separately as `npm run test:e2e`.
- CI does run it via the workflow file, so this is partially mitigated — but locally, `validate` can pass while the E2E suite is broken.

## LOW findings (13)

### L-1. Toast may cover walkthrough on mobile
- **File:** `src/components/Toast.tsx` lines ~29–32 (fixed bottom: 16px, center)
- On 375px viewport during a LineupDetail view, the toast can overlap the bottom row of the 2×2 walkthrough cards for 1.5s after copy.

- **Files:** `src/components/ScenarioCard.tsx` ~41–50, `src/components/CtPositionGuide.tsx` ~134–141

### L-3. Marker cluster key instability under data mutation
- **File:** `src/components/tabs/MapTab.tsx` ~59–65

### L-4. CLI tests miss atomic-write failure scenarios
- **File:** `scripts/new-lineup.test.mjs`, `new-scenario.test.mjs`

- **File:** `src/utils/coordinates.test.ts`

- **File:** `src/utils/steamDeepLink.test.ts`

### L-7. Heavy `waitForTimeout` use in E2E suite

- **Files:** `docs/USER_REQUIREMENTS.md` line ~100, `docs/DECISIONS_LEDGER.md` lines 154, 220

### L-11. SpawnPicker status `<div role="status">` isn't `aria-live`

- **File:** `src/components/TabBar.tsx`

### L-13. Focus management on view transitions
- **Files:** `src/App.tsx`, all detail components
- When entering ScenarioDetail or LineupDetail, focus stays on the clicked card. Should move to the new view's heading (h2) so keyboard / screen-reader users know the context changed.

## Audit methodology

Six parallel agents each took a focused lens:
1. State machine + reducer + navigation (reducer.ts, App.tsx, popstate, keydown, toast)
2. Data layer + boot validator (loadDust2.ts, dust2.json, types.ts, CLI scripts)
3. Component rendering + responsive CSS (all tab components, SpawnPicker, Radar, LineupDetail, etc.)
4. Test correctness + coverage (every *.test.* + *.spec.ts)
5. Doc consistency cross-check (URD, SOLUTION_DESIGN, ledger, brief, encyclopedia, refrag doc, README, user guide)
6. Accessibility + UX (keyboard nav, ARIA, focus, contrast, touch targets, reduced-motion)

Findings were merged, deduplicated, re-ranked, and spot-verified against actual file contents + command-line counts. Where an agent's claim was contradicted by verification (e.g. vitest count was 47 per grep, but actual is 77 per `vitest run`), the verified number wins.
